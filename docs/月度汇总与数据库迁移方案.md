## 库存管理系统 — 月度数据汇总 & Supabase 迁移方案

---

### 一、月度出入库数据汇总方案

参考主流库存管理系统（SAP MM、用友U8、金蝶K3）的通用做法，月度汇总建议分为"概览卡片 + 趋势图表 + 明细报表"三层结构。

#### 1. 时间区间筛选设计

在页面顶部放置一个统一的时间筛选栏，提供以下选项：

- **快捷按钮**：本月、上月、本季度、本年度
- **月份选择器**：`<input type="month">`，默认当前月
- **自定义区间**：两个日期选择器（开始日期 ~ 结束日期），支持跨年查询

筛选栏切换后，页面所有卡片和图表同步刷新。

#### 2. 概览卡片（KPI Cards）

页面顶部横排 6 张卡片：

| 卡片 | 数据来源 | 说明 |
|------|---------|------|
| 期初库存 | 上月末日 inventory 快照 | 显示物品数 + 总件数 |
| 本月入库 | stockInRecords 按日期筛选 | 入库笔数 + 总件数 |
| 本月出库 | stockOutRecords 按日期筛选 | 出库笔数 + 总件数 |
| 期末库存 | 期初 + 入库 - 出库 | 当前库存物品数 + 总件数 |
| 库存周转率 | 本月出库 / 平均库存 | 百分比，越高越好 |
| 低库存预警 | 低于安全库存的物品数 | 红色高亮 |

#### 3. 趋势图表

**出入库趋势折线图**：X 轴为日期（按天），双 Y 轴分别显示入库量和出库量。可以切换为"按周/按月"聚合。

**品类占比饼图**：按分类统计本月出库量，直观显示哪些品类消耗最多。

**月度环比柱状图**：展示近 6 个月的出库总量对比，一眼看出增减趋势。

#### 4. 明细报表（数据表格）

报表按"品类 → 物品"两级展开，每行显示：

| 列 | 说明 |
|----|------|
| 物品编号 | 唯一编码 |
| 物品名称 | - |
| 品类 | 分类名 |
| 期初库存 | 筛选期间开始时的库存 |
| 本期入库 | 该期间内入库合计 |
| 本期出库 | 该期间内出库合计 |
| 期末库存 | 筛选期间结束时的库存 |
| 变动率 | (期末-期初)/期初 × 100% |

支持按品类筛选、按变动率排序，右上角提供"导出 Excel"按钮。

#### 5. 页面结构建议

```
┌──────────────────────────────────────────────┐
│  [本月] [上月] [本季度] [本年度]   📅 月份选择  │
```

### 二、Supabase 数据库迁移方案

#### 1. 为什么需要迁移

当前系统使用 `localStorage` 存储数据，存在以下问题：

- **数据孤岛**：每台电脑/浏览器的数据互不相通，多人录入时数据不一致
- **容量限制**：localStorage 单域上限 5MB，出入库记录积累后可能不够用
- **无权限隔离**：无法在数据层面做行级安全控制（RLS）
- **无审计能力**：localStorage 无法追踪谁在什么时间修改了什么数据

#### 2. Supabase 选型优势

- **PostgreSQL**：关系型数据库，适合库存系统的强一致性需求
- **Row Level Security (RLS)**：可以根据用户角色控制数据访问范围
- **实时订阅 (Realtime)**：多人同时操作时，页面自动同步最新数据
- **Auth 内置**：用户注册/登录/角色管理开箱即用
- **免费额度**：500MB 数据库 + 1GB 存储 + 2GB 带宽，中小团队足够
- **RESTful API**：通过 `supabase-js` 客户端直接读写，无需写后端

#### 3. 数据库表设计

```sql
-- 用户表（扩展 Supabase Auth 的 auth.users）
CREATE TABLE profiles (
  id UUID REFERENCES auth.users PRIMARY KEY,
  name TEXT NOT NULL,
  role TEXT NOT NULL CHECK (role IN ('admin','purchase','warehouse','finance','staff')),
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 品类表
CREATE TABLE categories (
  id BIGSERIAL PRIMARY KEY,
  code TEXT UNIQUE NOT NULL,
  name TEXT NOT NULL,
  remark TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 库存物品表
CREATE TABLE inventory (
  id BIGSERIAL PRIMARY KEY,
  code TEXT UNIQUE NOT NULL,
  name TEXT NOT NULL,
  category_id BIGINT REFERENCES categories(id),
  brand TEXT,
  model TEXT,
  unit TEXT NOT NULL,
  stock INT DEFAULT 0,
  safety_stock INT DEFAULT 10,
  remark TEXT,
  updated_at TIMESTAMPTZ DEFAULT now()
);

-- 采购单表
CREATE TABLE purchase_orders (
  id BIGSERIAL PRIMARY KEY,
  code TEXT UNIQUE NOT NULL,
  purchase_date DATE NOT NULL,
  purchaser TEXT NOT NULL,
  supplier TEXT,
  total_amount DECIMAL(12,2),
  status TEXT DEFAULT 'pending',  -- pending → inbound_confirmed → completed
  remark TEXT,
  created_by UUID REFERENCES profiles(id),
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 采购单明细
CREATE TABLE purchase_items (
  id BIGSERIAL PRIMARY KEY,
  order_id BIGINT REFERENCES purchase_orders(id) ON DELETE CASCADE,
  item_name TEXT NOT NULL,
  brand TEXT,
  model TEXT,
  quantity INT NOT NULL,
  unit TEXT,
  unit_price DECIMAL(10,2),
  amount DECIMAL(12,2)
);

-- 入库记录表
CREATE TABLE stock_in_records (
  id BIGSERIAL PRIMARY KEY,
  code TEXT UNIQUE NOT NULL,
  purchase_order_id BIGINT REFERENCES purchase_orders(id),
  stock_in_date DATE NOT NULL,
  batch_no TEXT,
  status TEXT DEFAULT 'completed',
  confirmed_by UUID REFERENCES profiles(id),
  confirmed_at TIMESTAMPTZ DEFAULT now()
);

-- 入库明细
CREATE TABLE stock_in_items (
  id BIGSERIAL PRIMARY KEY,
  record_id BIGINT REFERENCES stock_in_records(id) ON DELETE CASCADE,
  inventory_id BIGINT REFERENCES inventory(id),
  quantity INT NOT NULL,
  unit_price DECIMAL(10,2)
);

-- 领用申请表
CREATE TABLE requisitions (
  id BIGSERIAL PRIMARY KEY,
  code TEXT UNIQUE NOT NULL,
  tour_date DATE,
  tour_name TEXT,
  scenario TEXT,
  applicant TEXT NOT NULL,
  apply_date DATE NOT NULL,
  total_quantity INT DEFAULT 0,
  status TEXT DEFAULT 'pending_outbound',
    -- pending_outbound → outbound_completed | cancelled
  remark TEXT,
  created_by UUID REFERENCES profiles(id),
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 领用明细
CREATE TABLE requisition_items (
  id BIGSERIAL PRIMARY KEY,
  requisition_id BIGINT REFERENCES requisitions(id) ON DELETE CASCADE,
  inventory_id BIGINT REFERENCES inventory(id),
  item_code TEXT,
  item_name TEXT,
  category TEXT,
  brand TEXT,
  model TEXT,
  unit TEXT,
  quantity INT NOT NULL
);

-- 出库记录表
CREATE TABLE stock_out_records (
  id BIGSERIAL PRIMARY KEY,
  code TEXT UNIQUE NOT NULL,
  requisition_id BIGINT REFERENCES requisitions(id),
  tour_date DATE,
  tour_name TEXT,
  scenario TEXT,
  stock_out_date DATE NOT NULL,
  total_quantity INT DEFAULT 0,
  status TEXT DEFAULT 'completed',
  confirmed_by UUID REFERENCES profiles(id),
  confirmed_at TIMESTAMPTZ DEFAULT now()
);

-- 出库明细
CREATE TABLE stock_out_items (
  id BIGSERIAL PRIMARY KEY,
  record_id BIGINT REFERENCES stock_out_records(id) ON DELETE CASCADE,
  inventory_id BIGINT REFERENCES inventory(id),
  item_code TEXT,
  item_name TEXT,
  requested_quantity INT,
  quantity INT NOT NULL,  -- 实际出库数量
  unit TEXT
);

-- 操作日志（审计）
CREATE TABLE audit_logs (
  id BIGSERIAL PRIMARY KEY,
  user_id UUID REFERENCES profiles(id),
  action TEXT NOT NULL,  -- create_requisition, confirm_stockout, etc.
  target_type TEXT,      -- requisition, inventory, etc.
  target_id BIGINT,
  detail JSONB,
  ip_address TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

#### 4. 行级安全策略 (RLS) 示例

```sql
-- 开启 RLS
ALTER TABLE requisitions ENABLE ROW LEVEL SECURITY;

-- 所有人可以看领用单
CREATE POLICY "所有人可读" ON requisitions
  FOR SELECT USING (true);

-- 只有 staff 和 admin 可以创建
CREATE POLICY "员工可创建" ON requisitions
  FOR INSERT WITH CHECK (
    auth.uid() IN (SELECT id FROM profiles WHERE role IN ('staff','admin'))
  );

-- 仓库角色可以更新状态为出库完成
CREATE POLICY "仓库可确认出库" ON requisitions
  FOR UPDATE USING (
    auth.uid() IN (SELECT id FROM profiles WHERE role IN ('warehouse','admin'))
  );
```

#### 5. 迁移路径（渐进式）

建议分三步迁移，不要一次性全改：

**第一步：搭建 Supabase 项目 + 数据同步层**
- 创建 Supabase 项目，建表
- 新建 `js/supabase-client.js`，初始化连接
- 新建 `js/db.js`，封装所有 CRUD 操作（与现有 localStorage API 同名函数）
- 在 `db.js` 内部做兼容：优先走 Supabase，失败时 fallback 到 localStorage

**第二步：替换各模块数据源**
- 按模块逐个替换：inventory → purchase → stockIn → requisition → stockOut
- 每替换一个模块，充分测试后再进入下一个
- 保留 localStorage 作为离线缓存（Supabase 断连时可用）

**第三步：接入 Supabase Auth + RLS**
- 替换现有的角色模拟登录为真实登录
- 配置 RLS 策略
- 添加 Realtime 订阅，实现多人实时同步

### 三、建议的优先级

1. **先做月度数据汇总**（纯前端，不改数据结构）— 用现有 localStorage 数据就能实现
2. **再做 Supabase 迁移**（底层改造）— 数据层替换，前端 UI 基本不变
3. **最后接入 Auth + Realtime** — 实现多人协作

这样的好处是每一步都可以独立上线验证，不会出现一次性大改导致的不稳定。
